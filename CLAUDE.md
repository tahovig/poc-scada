# Project: poc-scada (formerly scaffolded as `dpi-dnp3`)

## Goal

Third in a series of portfolio projects supporting a career pivot from software engineering to cybersecurity engineering (first: `poc-osint`, at `~/dev-projects/tah-osint-poc`; second: `poc-logids`, at `~/dev-projects/poc-logids`). A DNP3 protocol deep-packet-inspection (DPI) tool — analyzes captured SCADA/ICS network traffic (pcap files) and flags security-relevant patterns. Built around the user's professional background in critical infrastructure protection (energy generation/transmission/distribution, EMS) — the first project in the series where that specific domain expertise is the headline, not a footnote.

## Origin

Started independently by the user in late May/early June 2026 as `dpi-dnp3`. That repo's GitHub remote was never actually created — local commits existed with `origin` configured to point at `github.com/tahovig/dpi-dnp3`, but `git ls-remote`/`gh repo view` both confirmed nothing existed server-side (local `git status` reporting "up to date with origin" was just stale tracking metadata, not a live check). Picked back up and formalized from within the `poc-logids` session on 2026-07-18: since the old remote never existed, a fresh repo was created directly as `poc-scada` and the existing local commits pushed into it (rather than a true GitHub-side rename). Branch strategy applied (`main` + `develop`, matching `poc-osint`/`poc-logids`), local directory renamed from `dpi-dnp3` to `poc-scada` to match.

This was flagged as a stretch-goal candidate during `poc-logids`'s own data-source discussion (see that project's CLAUDE.md, "DNP3/ICS-SCADA" note) — DNP3/ICS analysis was set aside there specifically because `poc-logids` is deliberately log-based, not packet-capture-based, and DNP3 analysis is overwhelmingly a pcap-native discipline. Neither side knew about the other at the time: the `poc-logids` note assumed this would be a fresh idea, when the user had already independently started exactly this a month and a half earlier. This project is where it gets to be the headline.

## Decided so far

- **Type: pcap-native, offline/batch analysis of already-captured DNP3 traffic — not live packet capture.** Unlike `poc-logids` (which has a real live target: an internet-facing SSH honeypot), there's no equivalent live ICS network to tap here, so no live-tail/`-follow` analog is planned — scope is analyzing static pcap files. This also sidesteps the packet-capture privilege/environment concerns that ruled out live capture for `poc-logids` in the first place (root/`CAP_NET_RAW`, WSL2 networking) — nothing here needs elevated privileges or a live NIC, only file I/O over pcaps that already exist.
- **Git LFS already configured for `*.pcap` files** (carried over from the original `dpi-dnp3` commits, `.gitattributes`) — sensible as-is; pcaps can be large binary files that shouldn't bloat regular git history.
- **Language: Rust**, decided 2026-07-18. Confirmed over Python for the memory-safety-in-binary-parsing narrative described below. Installed via `rustup` (wasn't present on the machine) — stable toolchain, `~/.cargo/bin` on `PATH` after sourcing `~/.cargo/env`.
- **UI layer: Tauri v1 desktop app, built 2026-07-18, abandoned/reverted 2026-07-19.** Built a working desktop app (native window, file picker, findings table, IPC into `poc-scada-core`) with a source/destination traffic animation (SVG). The static parts — role-colored/labeled DNP3 address nodes (master/outstation, derived from traffic itself; real IP shown per address) — worked well and were confirmed via screenshots. The *animation* never did: this environment's WebKitGTK build (WSLg, running in a degraded EGL/software-rendering fallback visible as a startup warning) does not repaint SVG elements that JS mutates after insertion, under any technique tried — SMIL (`animateMotion`), `requestAnimationFrame`, `setTimeout`-driven attribute writes, opacity toggling. Confirmed via direct screenshot inspection across many isolated tests (no devtools access in this environment), including a `getBoundingClientRect()` readout proving the browser's own layout *was* updating correctly the whole time — nothing was ever actually composited to screen. The one thing that rendered anything at all was a CSS `@keyframes` animation on an element JS never touches again post-insertion, but even that only ever showed as a frozen mid-flight frame, not continuous motion, in this specific environment.
  - **User's call to abandon it (2026-07-19), not just a technical dead end** — two reasons given, both judged correct: (1) this was a genuine environment-specific rendering-pipeline fight, not a fixable design flaw, so further chasing it had bad ROI; (2) the animation was artificially *pacing* the replay (per-packet stagger/duration) purely for visual effect, which is actually contrary to what a DPI tool should do — report everything it found immediately, not act it out on a timer. Direction changed to: drop Tauri entirely, invest in making the CLI's terminal/ASCII output genuinely good instead (already a stated preference, see "Working preferences" below) — a return to this project's authentic style rather than a detour. All Tauri/`ui/` code removed (`git rm -rf src-tauri ui`), workspace membership and CI reverted to CLI-only (`ubuntu-latest`, no Node/webkit2gtk setup).
  - **Kept for the record in case a GUI is reconsidered later** — real Tauri v1 + Cargo-workspace footguns hit and fixed during the build: (1) `src-tauri/Cargo.toml` can't use `version.workspace = true` or `edition = "2024"` — `tauri-build`'s `cargo_toml` 0.15.3 dependency doesn't understand workspace inheritance and doesn't recognize the `"2024"` edition string. (2) `tauri.conf.json`'s `productName` must not collide with any Cargo binary name elsewhere in the workspace — `cargo tauri dev` copies the compiled app to a `productName`-derived path, not the Cargo bin name, and silently clobbered the CLI's compiled binary (both were `"poc-scada"`) on every dev run until renamed. (3) `beforeDevCommand`/`beforeBuildCommand` run with CWD = wherever `cargo tauri` was invoked from, *not* relative to `tauri.conf.json` like `devPath`/`distDir` are. (4) Vite's default port (5173) doesn't match Tauri's conventional `devPath` (`http://localhost:1420`) without an explicit `vite.config.ts` port pin. (5) This machine's Ubuntu 20.04 doesn't have `webkit2gtk-4.1` at all (confirmed via `apt-cache policy`) — Tauri v2 needs it, v1's `webkit2gtk-4.0` requirement was the only reason v1 was usable here at all; GitHub's `ubuntu-latest` (24.04) doesn't have `webkit2gtk-4.0` either, only `ubuntu-22.04` has both.
- **UI layer, current direction, decided 2026-07-19: richer terminal/ASCII CLI output**, replacing the Tauri attempt above. Not yet scoped/started — see "Next steps".
- **Detection scope for v1, decided 2026-07-18: both candidate checks**, implemented and passing against real fixture data:
  - Dangerous function codes: Cold Restart (13), Warm Restart (14), Direct Operate (5), Direct Operate No Ack (6).
  - Select-before-operate violations: an Operate (4) not immediately preceded by a Select (3) from the same DNP3 src/dest pair. (Direct Operate is intentionally *not* double-counted here — it bypasses select by design and is already caught by the dangerous-function-code check.)
- **Data source, decided 2026-07-18** after hands-on verification of two candidates (see below): **ITI `ICS-Security-Tools` DNP3 pcap collection is primary**, 4SICS is background noise only.
  - `github.com/ITI/ICS-Security-Tools/tree/master/pcaps/dnp3` — ~27 small, purpose-built, per-function-code pcaps (one function per file: `cold_restart_and_response.pcap`, `select_operate_and_responses.pcap`, `directoperate_and_response.pcap`, etc.). Verified by hand-decoding link/transport/application-layer bytes (no DNP3 crate exists, so this was done by directly parsing the frame structure) — filenames match actual frame contents. This is the primary source for both detections. Mirrored into `data/dnp3-iti/`.
  - Netresec's 4SICS Geek Lounge captures were checked **and mostly rejected**: of the three public files, one (`151020`) has zero DNP3 traffic (dominated by S7comm), one (`151021`) has port-20000 traffic that's actually scanner noise (Oracle TNS `CONNECT_DATA`, raw HTTP `GET /`), and only the third (`151022`) has genuine DNP3 frames — but just 36 "Request Link Status" link-layer probes with zero application-layer payload (no Select/Operate/Restart anywhere). Not usable as a detection-validation source. An 8,000-packet prefix of `151022` is kept in `data/4sics/` purely as a "realistic non-DNP3-heavy background" regression fixture (asserted to produce zero findings), not as DNP3 signal. Full verification methodology and rationale in `data/README.md`.
- **Large-scale real-world data source added, 2026-07-19: [qut-infosec/2017QUT_DNP3](https://github.com/qut-infosec/2017QUT_DNP3)** (QUT, 2017) — a genuinely different class of fixture from the small synthetic ITI captures: real, hours-long master/outstation polling traffic, with real attacker-injected commands in some categories. User explicitly asked for a large dataset to stress-test against, specifically requested pulling this after it surfaced (unverified) during the original 4SICS research.
  - Verified by running it through `poc-scada` itself, not just inspecting bytes: the **Control** category (`data/qut-2017/control/`) is a real-world negative control — 3.15M total packets, ~67K DNP3 messages, function codes exclusively `Confirm`/`Read`/`Response`/`UnsolicitedResponse` across all 4 training/testing × frequent/infrequent subsets checked, zero findings (correct, not a bug). The **Injection** category (`data/qut-2017/injection/`) contains genuine injected attacks — `slave.pcap`/`attacker.pcap` show real `ColdRestart`(36)/`WarmRestart`(42)/`ImmediateFreeze`(30) function codes; running `poc-scada` against `slave.pcap` produces **78 correct `dangerous-function-code` findings** with correct timestamps/flows/DNP3 addressing. First real (non-synthetic) confirmation of the detection logic against labeled attack traffic.
  - Almost picked the wrong category initially: `Control`'s consistent lack of dangerous function codes looked like it might be a detection gap, until the upstream tool's own attack-class listing (fetched from `Injection/Attack_script_output.txt`, a menu dump from an interrupted `DNP3_MITM_Lib` run) showed class names like `injection_ColdRestart`/`injection_WarmRestart` — confirming `Control` is baseline traffic and `Injection` was the actual match for this project's detections.
  - **Cost-aware about what actually gets committed**: upstream is ~1.36GB across 6 categories × training/testing × frequent/infrequent × 3 captures each. Only the `testing/frequent` subset of the two verified categories is kept (~702MB: ~175MB Control + ~527MB Injection) — deliberately under GitHub's free 1GB Git LFS storage cap (a real recurring-cost decision the user was asked about directly, not assumed) rather than the full raw pull. Caught and corrected a mistake mid-decision: initially quoted the *compressed zip* download sizes as if they were the actual uncompressed pcap sizes when presenting the size tradeoff — real uncompressed total is 702MB, not the ~568MB first quoted.
  - Not wired into `tests/integration.rs` — multi-hundred-MB real-world captures are for manual real-scale validation and demos, not something `cargo test` should run on every invocation. Full verification methodology in `data/README.md`.
  - **Follow-up, 2026-07-19: swept all 6 attack categories looking for `select-before-operate-violation` coverage, found none anywhere.** `Select`(3)/`Operate`(4) function codes never appear in Control, Injection, Replay, MITM, Flooding, or Masquerading — Replay/MITM checked at both `frequent`/`infrequent` granularity given that's where Injection's attack activity concentrated. Consistent across capture sessions spanning three different calendar months (Jul/Aug/Sep 2016), which points to this being a genuine property of the underlying testbed (it apparently never generates a select-then-operate control sequence at all, using direct write/freeze/restart actions instead) rather than something worth digging further for. Side effect of the sweep: also confirmed Replay/MITM/Flooding/Masquerading don't exercise `dangerous-function-code` either — only `Injection` does, across the whole dataset. Only `Control` was kept as a fixture (the negative-control story); the other four categories were inspected (function-code tallies only) and not committed, since they'd add redundant "no findings" coverage at real additional LFS cost — user explicitly chose "document only" over paying that cost for a fuller-but-redundant library. `select-before-operate-violation` coverage continues to come from the ITI fixtures + the synthetic bare-Operate test, as before.
- **Project structure, testing, CI, decided 2026-07-18:**
  - `src/lib.rs` exposes `pcap`, `dnp3` (`link`/`transport`/`application` submodules), and `detections` (one module per check) as a library; `src/main.rs` is a thin `clap`-based CLI over it. Library/binary split exists so `tests/integration.rs` can exercise the real parsing+detection path against fixture files.
  - Pcap reading: `pcap-parser` (pure Rust, no libpcap dependency — fits the offline/no-elevated-privileges framing). L2–L4 unwrapping: `etherparse`. DNP3 itself is hand-rolled (link/transport/application layers) — no mature Rust DNP3 crate exists, and this was the planned differentiator anyway.
  - Tests: unit tests colocated per module (parser edge cases, detection logic in isolation) plus `tests/integration.rs` running the full pcap → dnp3 → detections pipeline against the real fixtures in `data/`, including a synthetic in-memory pcap (built with `etherparse::PacketBuilder`, no fixture file needed) for the one violation pattern (bare Operate) that doesn't naturally occur in either public dataset.
  - CI: GitHub Actions (`.github/workflows/ci.yml`), matching `poc-logids`'s shape — `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo build`, `cargo test`. Checkout step uses `lfs: true` since fixtures are LFS-tracked.
  - `cargo clippy -- -D warnings` and `cargo fmt` both clean as of the initial scaffold; 24 tests passing (16 unit + 8 integration).

## Next steps

- **Algorithmic/complexity refactor tangent, started 2026-07-18.** User wants to explore genuine time/space complexity improvements to the parsing+analysis pipeline. First installment — the `link.rs` zero-copy refactor (user-authored proposal in `resources/poc-scada-refactor-01.txt`) — is **implemented**: `LinkFrame` now borrows (`&'a [u8]`) instead of allocating a `Vec<u8>` per frame, exposing user-data blocks via a lazy `user_data_chunks()` iterator that skips each block's 2-byte CRC without copying; frame length is computed in O(1) from the header's length byte (`div_ceil`) instead of by walking blocks. Reviewed before implementing: both core ideas were correct and worth adopting (traced the geometry formula against several `user_data_len` cases by hand, all matched); declined to carry over the proposal's "prevents LLVM vectorization" framing since it was unverified/likely overstated (`extend_from_slice` already lowers to `memcpy`, and the new code still bounds-checks each chunk slice at runtime) — the real, defensible wins are one fewer heap allocation per frame and an O(1) up-front validity check.
  - `transport.rs` now reads only the *first* user-data block (`LinkFrame::user_data_chunks().next()`) rather than a flat reassembled buffer — safe today because the transport header + application control byte + function code (user-data offsets 0–2) always fall inside DNP3's first 16-byte block, which is now stated explicitly in a doc comment rather than relied on implicitly. This will need real multi-block reassembly (copy-on-demand, not unconditional) the moment object-header/payload parsing is added — fixtures already exist for that (`file_read.pcap`, `file_write.pcap`).
  - Added 3 new `link.rs` unit tests that didn't exist before (zero-length user data, a real multi-block frame, a truncated second block) — the old implementation had no test coverage of the >16-byte-user-data path at all. 27 tests total now (up from 24), all passing; output verified byte-identical against the CLI on real fixtures before/after.
  - **Second installment, 2026-07-19 — pipeline-level streaming, also implemented.** `pcap::read_pcap` no longer returns `Vec<Packet>`; it returns `PcapPackets`, an iterator (`Item = Result<Packet, Error>`) that decodes one packet at a time as it's driven, instead of reading the whole capture into memory before analysis starts. Fuses after `Eof` or a hard parse error (`done: bool` flag) so a caller polling past an error (e.g. `filter_map(Result::ok)`) can't spin forever re-reading a broken stream. `dnp3::find_dnp3_messages` similarly changed from `Vec<Dnp3Message>` to `impl Iterator<Item = Dnp3Message> + '_` (built with `std::iter::from_fn`, same pattern as `link.rs`'s chunk iterator), removing the small per-packet `Vec` it used to allocate just to immediately drain. `lib.rs::analyze_pcap` now consumes both in a single pass, computing `packet_count` incrementally instead of via `.len()` on a materialized `Vec`. Both changes were fully internal — `read_pcap`/`find_dnp3_messages` had exactly one caller each (`analyze_pcap`; confirmed by grep before touching the signatures), so nothing in the CLI or Tauri app needed updating. Added `pcap.rs`'s first-ever unit tests (it had none before): streaming order, fuse-after-`Eof`, and immediate failure on a nonexistent path. 30 tests total now (up from 27); CLI output verified byte-identical on real fixtures before/after. `AnalysisReport.messages`/`.findings` are deliberately still materialized `Vec`s in the final report — both consumers (CLI printing, Tauri's one-shot IPC JSON payload) genuinely need "the whole report" as a unit, so streaming those further would be a consumer-facing redesign (event-based IPC, incremental CLI printing), not a parsing-pipeline complexity fix, and is out of scope for this tangent.
  - `Packet.payload` is still an owned `Vec<u8>` per packet (one small copy each) — structurally required by `pcap-parser`'s streaming internal buffer (recycled/refilled between `next()` calls, so a borrow can't safely outlive one iteration without an unsafe/callback-based redesign) and pre-dates this refactor. Noted as a known remaining allocation, not a regression, and not worth the redesign cost given it's already O(one packet) not O(file).
  - This series is now considered done for the moment — no further open items in it.
- **`direct_operate_crob_malform_but_good_crc.pcap`/`operate_aggressive_mode.pcap` edge cases, checked 2026-07-19: no gap found, closed out.** Both hand-decoded byte-by-byte and run through the real CLI. Both are `DirectOperate` (function 5, dest=4, src=3): the "malformed" fixture wraps a malformed CROB object; the "aggressive mode" one appends a DNP3 Secure Authentication MAC trailer directly after the object data in the same application fragment. Neither affects detection, and this isn't luck — the function code lives in the first 3 bytes of user data (transport header, app control byte, function code), always inside the first 16-byte link-layer block, well before any object-header/payload bytes this parser doesn't read at all. A malformed or extended object payload structurally can't hide a dangerous function code from a parser that never looks that far. Both fixtures now have explicit regression tests in `tests/integration.rs` (`direct_operate_with_malformed_crob_still_flags_dangerous_function_code`, `direct_operate_aggressive_mode_still_flags_dangerous_function_code`) — previously unexercised by any test. 32 tests total now (up from 30).
- **Terminal/ASCII CLI output improvements, decided and implemented 2026-07-19.** Replaces the Tauri UI direction (see "Decided so far"). Scope, as given by the user: (1) ANSI severity coloring — implemented as a two-tier rule-based split (not an arbitrary palette or a fabricated severity ranking beyond the two rules that already exist): red for `dangerous-function-code` (direct disruptive control actions), yellow for `select-before-operate-violation` (a real protocol-safety violation but not itself an active disruptive command); `crates/poc-scada-cli/src/color.rs`. (2) A simple animation *while processing* — explicitly not the same idea as the abandoned Tauri animation; this is a background-thread spinner (`crates/poc-scada-cli/src/spinner.rs`) that ticks for the duration of each file's `analyze_pcap` call and nothing more (no per-packet progress — `analyze_pcap` doesn't expose incremental progress, and adding that API surface for a batch CLI tool wasn't judged worth it). Pure ANSI cursor control in a real terminal — a completely different, well-trodden mechanism from the SVG/webview path that broke, with none of the same risk. (3) Logical categorization behind an optional flag — `--group-by-rule`, default off (chronological order unchanged); when set, findings are grouped under a heading per rule instead of listed flat. Both color and the spinner auto-disable for non-terminal (piped/redirected) output via `std::io::IsTerminal`, matching `poc-logids`'s established policy. Verified color/spinner activation under a real TTY using `script -qec ... /dev/null` to fake one (this sandboxed environment's own terminal isn't a real TTY) — confirmed padding happens *before* ANSI wrapping so column alignment survives having color on, a classic gotcha with fixed-width formatting + escape codes. All existing tests unaffected (they exercise `analyze_pcap` directly, not the CLI's text rendering) — 30 tests still passing.
