# Project: poc-scada (formerly scaffolded as `dpi-dnp3`)

## Goal

Third in a series of portfolio projects supporting a career pivot from software engineering to cybersecurity engineering (first: `poc-osint`, at `~/dev-projects/tah-osint-poc`; second: `poc-logids`, at `~/dev-projects/poc-logids`). A DNP3 protocol deep-packet-inspection (DPI) tool — analyzes captured SCADA/ICS network traffic (pcap files) and flags security-relevant patterns. Built around the user's professional background in critical infrastructure protection (energy generation/transmission/distribution, EMS) — the first project in the series where that specific domain expertise is the headline, not a footnote.

## Origin

Started independently by the user in late May/early June 2026 as `dpi-dnp3`. That repo's GitHub remote was never actually created — local commits existed with `origin` configured to point at `github.com/tahovig/dpi-dnp3`, but `git ls-remote`/`gh repo view` both confirmed nothing existed server-side (local `git status` reporting "up to date with origin" was just stale tracking metadata, not a live check). Picked back up and formalized from within the `poc-logids` session on 2026-07-18: since the old remote never existed, a fresh repo was created directly as `poc-scada` and the existing local commits pushed into it (rather than a true GitHub-side rename). Branch strategy applied (`main` + `develop`, matching `poc-osint`/`poc-logids`), local directory renamed from `dpi-dnp3` to `poc-scada` to match.

This was flagged as a stretch-goal candidate during `poc-logids`'s own data-source discussion (see that project's CLAUDE.md, "DNP3/ICS-SCADA" note) — DNP3/ICS analysis was set aside there specifically because `poc-logids` is deliberately log-based, not packet-capture-based, and DNP3 analysis is overwhelmingly a pcap-native discipline. Neither side knew about the other at the time: the `poc-logids` note assumed this would be a fresh idea, when the user had already independently started exactly this a month and a half earlier. This project is where it gets to be the headline.

## Decided so far

- **Type: pcap-native, offline/batch analysis of already-captured DNP3 traffic — not live packet capture.** Unlike `poc-logids` (which has a real live target: an internet-facing SSH honeypot), there's no equivalent live ICS network to tap here, so no live-tail/`-follow` analog is planned — scope is analyzing static pcap files. This also sidesteps the packet-capture privilege/environment concerns that ruled out live capture for `poc-logids` in the first place (root/`CAP_NET_RAW`, WSL2 networking) — nothing here needs elevated privileges or a live NIC, only file I/O over pcaps that already exist.
- **Git LFS already configured for `*.pcap` files** (carried over from the original `dpi-dnp3` commits, `.gitattributes`) — sensible as-is; pcaps can be large binary files that shouldn't bloat regular git history.
- **Language: Rust**, decided 2026-07-18. Confirmed over Python for the memory-safety-in-binary-parsing narrative described below. Installed via `rustup` (wasn't present on the machine) — stable toolchain, `~/.cargo/bin` on `PATH` after sourcing `~/.cargo/env`.
- **UI layer: Tauri v1 desktop app, decided and implemented 2026-07-18.** A source/destination traffic animation (plain SVG + `<animateMotion>`, `ui/src/animation.ts`) replays every DNP3 message as a packet flying between labeled address nodes, red for anything a detection flagged, alongside the same findings table the CLI prints. Frontend: Vite + vanilla TypeScript (no framework) in `ui/`, deliberately minimal since the user was new to Rust↔frontend integration and wanted to learn the pattern, not a JS framework. Verified actually rendering via a real WSLg GUI window (screenshotted through PowerShell interop, since no Linux-side screenshot tool was available) with real fixture data flowing through end-to-end — not just "it compiles."
  - **Tauri v1, not v2**: this machine's Ubuntu 20.04 genuinely doesn't have `webkit2gtk-4.1` available in its repos (confirmed via `apt-cache policy`, not assumed) — v2 requires it, v1 uses `webkit2gtk-4.0` which is available. User explicitly chose to stay on v1 rather than upgrade the WSL2 distro. System deps (`libwebkit2gtk-4.0-dev build-essential curl wget file libssl-dev libgtk-3-dev libayatana-appindicator3-dev librsvg2-dev`) needed a `sudo apt install` the user ran themselves (no passwordless sudo in this environment). Node.js installed via `nvm` (`~/.nvm`) since only a Windows `node.exe` was reachable via WSL interop, which is unreliable for this workflow — must `source ~/.nvm/nvm.sh` before any `npm`/`node`/`cargo tauri` command that needs Node.
  - **Repo restructured into a Cargo workspace** to support this: `crates/poc-scada-core` (the actual DPI logic, GUI/CLI-agnostic — `pcap`, `dnp3`, `detections`, plus a new shared `analyze_pcap` pipeline fn), `crates/poc-scada-cli` (thin CLI over it, binary name still `poc-scada`), `src-tauri` (Tauri backend, one `#[tauri::command]` calling the same `analyze_pcap`). Old root `src/`/`tests/` moved via `git mv` to preserve history.
  - **Several Tauri v1 + workspace footguns hit and fixed, worth remembering if this breaks again:** (1) `src-tauri/Cargo.toml` can't use `version.workspace = true` or `edition = "2024"` — `tauri-build`'s `cargo_toml` 0.15.3 dependency doesn't understand workspace inheritance and doesn't recognize the `"2024"` edition string; both hardcoded there only, edition `2021`, which is fine since editions can differ per-crate in one workspace. (2) `tauri.conf.json`'s `productName` must not collide with any Cargo binary name elsewhere in the workspace — `cargo tauri dev` copies the compiled app to a `productName`-derived path, not the Cargo bin name, and it was originally `"poc-scada"`, silently clobbering the CLI's compiled binary of the same name on every dev run. Renamed to `"poc-scada-desktop"`. (3) `beforeDevCommand`/`beforeBuildCommand` run with CWD = wherever `cargo tauri` was invoked from (the repo root, by convention here), *not* relative to `tauri.conf.json` like `devPath`/`distDir` are — so they're `npm --prefix ui run ...`, not `npm --prefix ../ui run ...`. (4) Vite's default port (5173) doesn't match Tauri's conventional `devPath` (`http://localhost:1420`) — needed an explicit `ui/vite.config.ts` pinning `server.port = 1420`.
- **Detection scope for v1, decided 2026-07-18: both candidate checks**, implemented and passing against real fixture data:
  - Dangerous function codes: Cold Restart (13), Warm Restart (14), Direct Operate (5), Direct Operate No Ack (6).
  - Select-before-operate violations: an Operate (4) not immediately preceded by a Select (3) from the same DNP3 src/dest pair. (Direct Operate is intentionally *not* double-counted here — it bypasses select by design and is already caught by the dangerous-function-code check.)
- **Data source, decided 2026-07-18** after hands-on verification of two candidates (see below): **ITI `ICS-Security-Tools` DNP3 pcap collection is primary**, 4SICS is background noise only.
  - `github.com/ITI/ICS-Security-Tools/tree/master/pcaps/dnp3` — ~27 small, purpose-built, per-function-code pcaps (one function per file: `cold_restart_and_response.pcap`, `select_operate_and_responses.pcap`, `directoperate_and_response.pcap`, etc.). Verified by hand-decoding link/transport/application-layer bytes (no DNP3 crate exists, so this was done by directly parsing the frame structure) — filenames match actual frame contents. This is the primary source for both detections. Mirrored into `data/dnp3-iti/`.
  - Netresec's 4SICS Geek Lounge captures were checked **and mostly rejected**: of the three public files, one (`151020`) has zero DNP3 traffic (dominated by S7comm), one (`151021`) has port-20000 traffic that's actually scanner noise (Oracle TNS `CONNECT_DATA`, raw HTTP `GET /`), and only the third (`151022`) has genuine DNP3 frames — but just 36 "Request Link Status" link-layer probes with zero application-layer payload (no Select/Operate/Restart anywhere). Not usable as a detection-validation source. An 8,000-packet prefix of `151022` is kept in `data/4sics/` purely as a "realistic non-DNP3-heavy background" regression fixture (asserted to produce zero findings), not as DNP3 signal. Full verification methodology and rationale in `data/README.md`.
- **Project structure, testing, CI, decided 2026-07-18:**
  - `src/lib.rs` exposes `pcap`, `dnp3` (`link`/`transport`/`application` submodules), and `detections` (one module per check) as a library; `src/main.rs` is a thin `clap`-based CLI over it. Library/binary split exists so `tests/integration.rs` can exercise the real parsing+detection path against fixture files.
  - Pcap reading: `pcap-parser` (pure Rust, no libpcap dependency — fits the offline/no-elevated-privileges framing). L2–L4 unwrapping: `etherparse`. DNP3 itself is hand-rolled (link/transport/application layers) — no mature Rust DNP3 crate exists, and this was the planned differentiator anyway.
  - Tests: unit tests colocated per module (parser edge cases, detection logic in isolation) plus `tests/integration.rs` running the full pcap → dnp3 → detections pipeline against the real fixtures in `data/`, including a synthetic in-memory pcap (built with `etherparse::PacketBuilder`, no fixture file needed) for the one violation pattern (bare Operate) that doesn't naturally occur in either public dataset.
  - CI: GitHub Actions (`.github/workflows/ci.yml`), matching `poc-logids`'s shape — `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo build`, `cargo test`. Checkout step uses `lfs: true` since fixtures are LFS-tracked.
  - `cargo clippy -- -D warnings` and `cargo fmt` both clean as of the initial scaffold; 24 tests passing (16 unit + 8 integration).

## Next steps

- **Algorithmic/complexity refactor tangent, started 2026-07-18.** User wants to explore genuine time/space complexity improvements to the parsing+analysis pipeline. First installment — the `link.rs` zero-copy refactor (user-authored proposal in `resources/poc-scada-refactor-01.txt`) — is **implemented**: `LinkFrame` now borrows (`&'a [u8]`) instead of allocating a `Vec<u8>` per frame, exposing user-data blocks via a lazy `user_data_chunks()` iterator that skips each block's 2-byte CRC without copying; frame length is computed in O(1) from the header's length byte (`div_ceil`) instead of by walking blocks. Reviewed before implementing: both core ideas were correct and worth adopting (traced the geometry formula against several `user_data_len` cases by hand, all matched); declined to carry over the proposal's "prevents LLVM vectorization" framing since it was unverified/likely overstated (`extend_from_slice` already lowers to `memcpy`, and the new code still bounds-checks each chunk slice at runtime) — the real, defensible wins are one fewer heap allocation per frame and an O(1) up-front validity check.
  - `transport.rs` now reads only the *first* user-data block (`LinkFrame::user_data_chunks().next()`) rather than a flat reassembled buffer — safe today because the transport header + application control byte + function code (user-data offsets 0–2) always fall inside DNP3's first 16-byte block, which is now stated explicitly in a doc comment rather than relied on implicitly. This will need real multi-block reassembly (copy-on-demand, not unconditional) the moment object-header/payload parsing is added — fixtures already exist for that (`file_read.pcap`, `file_write.pcap`).
  - Added 3 new `link.rs` unit tests that didn't exist before (zero-length user data, a real multi-block frame, a truncated second block) — the old implementation had no test coverage of the >16-byte-user-data path at all. 27 tests total now (up from 24), all passing; output verified byte-identical against the CLI on real fixtures before/after.
  - **Second installment, 2026-07-19 — pipeline-level streaming, also implemented.** `pcap::read_pcap` no longer returns `Vec<Packet>`; it returns `PcapPackets`, an iterator (`Item = Result<Packet, Error>`) that decodes one packet at a time as it's driven, instead of reading the whole capture into memory before analysis starts. Fuses after `Eof` or a hard parse error (`done: bool` flag) so a caller polling past an error (e.g. `filter_map(Result::ok)`) can't spin forever re-reading a broken stream. `dnp3::find_dnp3_messages` similarly changed from `Vec<Dnp3Message>` to `impl Iterator<Item = Dnp3Message> + '_` (built with `std::iter::from_fn`, same pattern as `link.rs`'s chunk iterator), removing the small per-packet `Vec` it used to allocate just to immediately drain. `lib.rs::analyze_pcap` now consumes both in a single pass, computing `packet_count` incrementally instead of via `.len()` on a materialized `Vec`. Both changes were fully internal — `read_pcap`/`find_dnp3_messages` had exactly one caller each (`analyze_pcap`; confirmed by grep before touching the signatures), so nothing in the CLI or Tauri app needed updating. Added `pcap.rs`'s first-ever unit tests (it had none before): streaming order, fuse-after-`Eof`, and immediate failure on a nonexistent path. 30 tests total now (up from 27); CLI output verified byte-identical on real fixtures before/after. `AnalysisReport.messages`/`.findings` are deliberately still materialized `Vec`s in the final report — both consumers (CLI printing, Tauri's one-shot IPC JSON payload) genuinely need "the whole report" as a unit, so streaming those further would be a consumer-facing redesign (event-based IPC, incremental CLI printing), not a parsing-pipeline complexity fix, and is out of scope for this tangent.
  - `Packet.payload` is still an owned `Vec<u8>` per packet (one small copy each) — structurally required by `pcap-parser`'s streaming internal buffer (recycled/refilled between `next()` calls, so a borrow can't safely outlive one iteration without an unsafe/callback-based redesign) and pre-dates this refactor. Noted as a known remaining allocation, not a regression, and not worth the redesign cost given it's already O(one packet) not O(file).
  - This series is now considered done for the moment — no further open items in it.
- Consider whether the two v1 detections need refinement once more traffic patterns are explored (e.g. `direct_operate_crob_malform_but_good_crc.pcap` and `operate_aggressive_mode.pcap` in the ITI set are unexplored edge cases).
